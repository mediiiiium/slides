なぜ卒業？ · チャット vs ファイル

チャットは毎回リセット、ファイルは積み上がる

同じAIでも、成果を“どこに置くか”で伸び方がまるで違う。
Before · チャットだけ

毎回ふりだし

会話を閉じると文脈は消える。次はまた一から説明。成果は“流れて”いって、手元には何も残らない。

After · ファイルを持つ

昨日の続きから

決めたこと・作ったものがフォルダに残る。「このファイル読んで」で再開でき、日々の作業が“資産”になっていく。

卒業とは——成果をチャットの履歴から、自分のフォルダへ移すこと。

軽量にサクサク、パワフルに動かす。慣れてきたら、コマンドで開発・自動化をがっつり回せる。動くのはアプリの Code と同じもの。

デスクトップアプリ まずこっち

チャットだけでなく、Code も Design も全部入り。まず開くべきは Code タブ——ファイルを触るのはここ。

本命は、その中の Code でファイルを触ること——ここがチャット卒業の一手。

⚙️ 技術メモ

入り口は違っても、動かしているのは同じ Claude。Cowork／Code は手元のファイルやCLIを、Design はブラウザ上のキャンバスを扱う。Windows でデスクトップアプリの Code を初めて開くときは Git for Windows が必要（入れて再起動）。

使いこなしのコツ · 文脈の残し方

文脈は、ファイルに残す

AIは前回の会話を覚えていない。だから経緯をファイルに残して渡す。名前は少しこわいだけで、やることはシンプル。
残す

フォルダに残す

作業の経過を .md でフォルダに。次のセッションや人へ、そのまま引き継げる。

履歴 · commit

git＝セーブ点

区切りのいいところで記録すれば、いつでも前の状態に戻せる。ゲームのセーブポイント。

共有 · push

GitHub＝クラウド

git の記録をクラウドへ。上げておけば別のPC・人と共有でき、バックアップにもなる。

迷ったら「このファイル読んで」。.md 一枚で、前提を丸ごと渡せる。

⚙️ 技術メモ

操作は全部 Claude（Code）に任せられる——「ここまでコミットして」「GitHub に上げて」で通じる。会話が長くなったら compact で畳んで継続。実際、下書き管理は Notion から GitHub に移した。仕組みを完璧に覚えなくてOK。

使いこなしのコツ · 考え方 ②

プロンプト → ループ設計へ

“うまく頼む”から、“うまく回す”へ。

期待する振る舞いやアウトプットは、まずプロンプト（指示）で伝える。ただ最近は“一発のプロンプト”より、AI に試す→確認→直すを繰り返させる「ループの設計」へ、ステージが移ってきている。

試す→ 確認→ 直す↻ くり返す

指示は“ゴール”を渡すだけ。あとは AI が回して近づける。

⚙️ 技術メモ

いわゆるエージェントループ。実行→出力をフィードバック（テスト・エラー・レビュー）で受け、AI 自身が直して再実行する。プロンプト職人芸から、良し悪しを測る評価（eval）とループ設計へ重心が移りつつある。

業務の自動化から、子どもの漢字ドリルまで——大きな話ばかりじゃない。ここまでのコツ（文脈を残す・区切る・ループで回す）が、こう形になる。コードの書けない裏方が、失敗しながら、全部ひとりで。
CASE 01 · 配達アプリ
配達ルート、もう頭で悩まない。

ビール樽を飲食店に配達。回る順番を毎回手で考えていた → 自動で最適ルートに。

依頼メッセージ→ 店名を抽出→ Google Mapsで最短ルート→ 配達ログ→ LINE通知
  • 今：出発前にスマホで店名を入れる → 順番と所要時間が出る。準備のストレスが消えた。
  • おまけ：GitHubにAPIキーを晒して大炎上 → Claudeに聞いて数分で対処。不正利用はゼロで済んだ。
⚙️ 技術メモ

店名抽出は Claude API。ルートは最近傍法＋2optで計算（GAS＋Google Maps API）。キーは ScriptProperties に退避——直書き厳禁を身をもって学んだ。

CASE 02 · Podcast自動編集
深夜3時間の編集が、ビール片手に“待つだけ”に。
マイク3本の生音→ ノイズ除去→ 音量正規化→ イントロ/アウトロ結合→ 書き出し
  • コマンド一発で、文字起こしまで自動。走らせて放っておけば仕上がっている。
  • 「始まりました」を探して、冒頭の雑談を自動でカットする小技も後から追加。
⚙️ 技術メモ

pipeline.py（実装は Claude Code）。Auphonic API＋ffmpeg（2秒クロスフェード）＋faster-whisper。※初回はサンプルレートのズレで再生速度がバグり、全員ミニオンの声になった。

CASE 03 · Now on Tap
「今このバーで何が飲めるか」を、歩く前に。

店のInstagramを毎回見るのが面倒 → ロボットに毎朝見回らせて、東京のビールDBにする。コンセプトは Walk in knowing.

Instagram取得→ Vision AIでビール情報をパース→ データ整備→ DB化
  • 現実：AIが「ハートランド」をビアスタイルと誤読（91件）…。きれいに集めるより、データ掃除が本番だった。
⚙️ 技術メモ

Instagramは難読化対策でUIでなくデータ取得。GitHub Actionsで毎朝5時に巡回。表記ゆれは“決定的マップ”で固定し、初見の難物だけAIに投げる——ゆらぎ判断を毎回AIに投げると、逆にゆらぎが増える。

その他 · OSSへの挑戦

非エンジニアの、OSSへの挑戦

日本のSaaSを、AIに直接つなぐ — mcp-jp

まず MCP とは：AIを外部のツールやデータに繋ぐための“共通プラグイン規格”。対応させると、AI がそのサービスを直接読み書きできるようになる。

  • 日本のSaaS（freee・勤怠・採用…）を MCP で AI に直結。現在34サービス。
  • 公式コネクタが出ていない“空白地帯”を、非エンジニアが自分で作って埋める。
  • しかも自分用で終わらせず、誰でも使える形で公開——「無ければ作って、みんなに配る」OSSの発想。

コードは書けなくても、“無ければ作って公開する”側に回れる。

⚙️ 技術メモ

MCP＝Model Context Protocol。AI と外部ツールの間の共通規格で、一度対応すればどの対応AIからも同じように叩ける。mcp-jp は各SaaSのAPIを MCPサーバー化して束ねたもの——公式が動くまでの“橋渡し”を、コミュニティ側で先に用意しておくイメージ。
